Code Comments

One of the most important lessons you can learn about writing code is that it's not just for the computer. Code is every bit as much, if not more, for the developer as it is for the compiler.

You should strive not just to write programs that work correctly, but programs that make sense when examined. Code commenting is an imporatant part of that.

Some observations and guidelines to good commenting

  • Code without commenting is suboptimal.
  • Too many comments (one per line, for example) is probably a sign of poorly written code.
  • Comments should explain why, not what. They can optionally explain how if that's particularly confusing.

There are two types of comments possible.

Consider:

// This is a single line comment.

/* But this is
a multiline
comment. */

the // single-line comment is appropriate if you're going to put a comment right above a single statement, or even at the end of a line. Usually short.

the /*..*/ comment is appropriate if you have several lines worth of explanation. It can appear anywhere on a line, even in the middle of a line.
